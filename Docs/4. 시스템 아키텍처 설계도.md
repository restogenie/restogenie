# **메타데이터 주도 다중 테넌트 데이터 파이프라인 아키텍처 (AWS 환경)**

## **1\. 전체 아키텍처 개요**

본 파이프라인은 서버리스 컴퓨팅 자원을 공유(Pooled)하되 데이터 계층은 격리하는 하이브리드 다중 테넌트 아키텍처를 따르며, Bronze-Silver-Gold 데이터 레이크 패턴으로 구성됩니다. 하드코딩 없이 사용자가 정의한 'JOLT Spec' 메타데이터에 의해 Transformation 엔진이 구동됩니다.

## **2\. 데이터 생애주기 (Data Lifecycle) 및 AWS 컴포넌트**

### **Phase 1: 스케줄링 및 트리거 (Triggering)**

* **컴포넌트:** Amazon EventBridge, Amazon RDS (Metadata DB), AWS KMS  
* **로직:**  
  1. EventBridge Cron에 의해 테넌트별 동기화 워크플로우 트리거.  
  2. 핸들러가 RDS(pos\_connections 테이블)에서 대상 매장의 POS API 엔드포인트와 KMS로 암호화된 API Key 조회.

### **Phase 2: 추출 및 원시 데이터 적재 (Extraction & Bronze Layer)**

* **컴포넌트:** AWS Lambda (또는 ECS Fargate), Amazon S3 (Bronze Bucket)  
* **로직:**  
  1. Worker(Lambda)가 POS API 호출하여 JSON 페이로드 다운로드.  
  2. 서드파티 API 에러 대응: tenacity 라이브러리 활용 \-\> **Exponential Backoff with Jitter** 적용 (최대 3\~5회 재시도, 일시적 오류 429/500에만 적용).  
  3. 다운로드된 Raw JSON을 변형 없이 S3 Bronze 구역에 적재 (경로 예: s3://rg-bronze/tenant\_id=101/date=20260225/raw\_pos\_data.json).

### **Phase 3: 선언적 데이터 변환 (Transformation & Silver Layer)**

* **컴포넌트:** AWS Glue (또는 EMR/Lambda), JOLT 라이브러리, Amazon RDS (data\_mappings 테이블)  
* **로직:**  
  1. S3 객체 생성 이벤트가 변환 엔진(Glue) 트리거.  
  2. RDS에서 해당 테넌트가 UI에서 설정한 **JOLT Spec JSON**을 메모리에 로드.  
  3. Raw JSON \+ JOLT Spec을 JOLT 엔진에 주입하여 레스토지니 표준 스키마(ARTS ODM 기반)로 구조 평탄화(Flattening) 및 키 이동(Shift).  
  4. **Per-row Error Handling:** 타입 불일치 등 파싱 실패 시 파이프라인을 중단하지 않고, 해당 행의 \_restogenie\_meta (JSONB) 필드에 에러 내역 기록 후 강제 적재.

### **Phase 4: 최종 적재 (Loading & Target Sales DB \- Gold Layer)**

* **컴포넌트:** Amazon Aurora PostgreSQL (또는 Snowflake)  
* **로직:**  
  1. JOLT 변환을 통과한 무결한 데이터가 통합 매출 데이터베이스 표준 테이블에 병합(Upsert/Merge).  
  2. 테넌트 간 Noisy Neighbor 방지를 위해 Row-Level Security(RLS) 적용.

## **3\. 핵심 기술 요소**

* **동적 스키마 추론 엔진 (Backend):** API 호출 \-\> JSON 루트\~리프 노드 재귀 탐색 \-\> 경로 평탄화 \-\> 가장 넓은 타입(Widest Type) 캐스팅 알고리즘.  
* **JOLT (JSON Language for Transform):** 코드가 아닌 설정(Config) 데이터만으로 JSON을 변환하는 오픈소스 프레임워크. (Operation: shift, default, remove 사용)
